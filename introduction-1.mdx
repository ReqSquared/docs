We've seen this every week. Reqs piling up. Recruiters running between meetings. Some chairs empty. Some chairs buried.

Standard ATS reporting shows you the counts. Number of reqs per recruiter. Average per team. Distribution by region. Useful for executive summaries. Insufficient for the actual decision: _which req should move, to which recruiter, right now_.

Load balancing as a BlueCube is the methodology that produces that decision.

The cube reads the current open req inventory. The cube reads each recruiter's portfolio shape — what they're carrying, what difficulty cluster their reqs fall into, where they sit on the efficiency frontier. The cube applies TA Context — priority weights, severity thresholds, capacity ceilings — to compose a routing decision per req.

Output is a recommendation. _This req moves to this recruiter. This recruiter has capacity. This req's difficulty pattern matches this recruiter's specialization._ Each recommendation traceable to the inputs that produced it.

The pipeline meeting changes. Instead of _who has time_, the question becomes _what does the methodology say_. The team decides whether to follow the recommendation. The methodology produces the structured starting point.

Load balancing as a cube. Routing as the output. Pipeline meetings get faster. Reqs stop sitting on overloaded recruiters while empty chairs sit elsewhere.
