A single requisition gets compared across every context lens defined in the config. How many lenses qualify depends on the data.

If a groupby combination meets the configured `top_n` sample threshold, it becomes a valid context lens. If it does not, the engine moves to the next broader grouping.

## The Lens Hierarchy

Context lenses are defined in `metric_config.yml`. Each lens group has its own `top_n` sample threshold and a hierarchy of progressively broader groupings.

```yaml
# metric_config.yml

req_iq_cluster_1:

  recruiter_context:
    top_n: 42
    lineage_columns:
      - ["recruiter", "country", "job_group", "job_family", "job_level"]  # most specific
      - ["recruiter", "country", "job_group", "job_family"]
      - ["recruiter", "country", "job_group"]
      - ["recruiter", "country"]
      - ["recruiter"]                                                     # broadest

  country_context:
    top_n: 240
    lineage_columns:
      - ["country", "job_group", "job_family", "job_level"]
      - ["country", "job_group", "job_family"]
      - ["country", "job_group"]
      - ["country"]

  job_group_context:
    top_n: 480
    lineage_columns:
      - ["job_group", "job_family", "job_level"]
      - ["job_group", "job_family"]
      - ["job_group"]
```

Three context groups. Each with its own `top_n`. Each with a hierarchy from most specific to broadest.

The `top_n` differs by group because the analytical question differs:

- **Recruiter context** uses `top_n: 42` — small enough to surface individual recruiter patterns, large enough to be statistically valid
- **Country context** uses `top_n: 240` — accounts for the larger population pool at country grain
- **Job group context** uses `top_n: 480` — accounts for the broader cross-organizational comparison

## How Lenses Qualify

The engine evaluates each context lens against the population data. A lens qualifies if the groupby combination produces at least `top_n` records.

For a given requisition, the number of qualifying context lenses varies based on:

- The recruiter's historical volume in this country × job_group × job_family × job_level combination
- The country's historical volume across job categories
- The job group's historical volume

A new recruiter in a thin-data country might only qualify for the broadest lenses. A senior recruiter in a high-volume function might qualify for every lens in the hierarchy.

## What Each Lens Tells You

The z-score tells you how this req compares to that specific peer group.

A req can be:

- *Faster than this recruiter's normal* — recruiter context, z-score below zero on time-to-fill
- *Slower than the country baseline* — country context, z-score above zero on time-to-fill
- *Average against the company baseline* — job group context, z-score near zero

All three are true simultaneously. Each tells a different operational story:

- Recruiter context surfaces the recruiter's individual performance pattern
- Country context surfaces the regional pattern
- Job group context surfaces the cross-org function pattern

## Primary and Secondary Benchmarks

Within each context group, the most specific qualifying lens is the **primary benchmark**. Broader lenses in the same group are **secondary benchmarks**.

For recruiter context:

- Primary: most specific qualifying lens (e.g., recruiter × country × job_group × job_family × job_level if `top_n` met)
- Secondary: broader lenses in the recruiter context group

Primary benchmarks drive decisions. Secondary benchmarks provide stability and confirmation. The engine surfaces both, separated, so operators see which lens is the decision-relevant one and which are providing context.

## The Variable Output

The number of valid context lenses per requisition varies entirely based on the data meeting the configured thresholds.

A requisition with rich peer group history across every dimension might score against 7-10 context lenses. A requisition in a niche function or new geography might score against only 2-3.

The engine never fabricates context. If the peer group sample doesn't exist at the required threshold, that lens is dropped for that requisition. The output is honest about what could be measured and what couldn't.

## The Thesis

Same requisition. Different context lens. Different signal.

Standard reporting collapses each requisition into a single comparison — usually a company-wide average. The Descriptive System refuses to collapse. Every qualifying lens produces its own signal. The composite picture across lenses is what makes the system descriptive.

The engine doesn't decide which lens matters most. The operator does. The engine provides the multi-lens evidence.
