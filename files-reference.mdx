The codebase is organized by responsibility. Pipeline orchestration, pipeline functions, shared ETL, configuration loading, schemas, and configs each live in their own file.

## Code Files

| File | Role |
|------|------|
| `requisition_iq.py` | Pipeline orchestrator. Runs steps 1 through 5. |
| `iq.py` | Pipeline functions. All statistical logic lives here. |
| `etl_functions.py` | Shared ETL functions. Lineage columns, groupby, z-scores, survival bins. |
| `data_importer.py` | Config loader. Reads YAML, loads CSVs, applies data types, resolves pipeline configs. |

## Configuration Files

| File | Role |
|------|------|
| `config/schema_filled_reqs.yml` | Column schema with data types, descriptions, z-score thresholds, direction, floor/ceiling. |
| `config/metric_config.yml` | Context lens definitions (which groupby combinations to run) and global metric selections. |
| `config/dataframe.yml` | Pipeline configuration. Which data source, which quarter is observation vs population, which schema and cluster config to use. |

## requisition_iq.py

The orchestrator. Reads the configuration, runs the five pipeline steps, produces the named output dataframes.

Top of the file loads the config:

```python
config = dimp.get_dataframe_config("filled_req_iq_cluster_1_df")

observations_df = config["observations_df"]
population_df = config["population_df"]
schema = config["schema"]
unique_id = config["unique_id"]
global_metrics_selected = config["global_metrics_selected"]
cluster_list_of_lists = config["cluster_list_of_lists"]
sort_by_for_top_n = config["sort_by_for_top_n"]
```

The rest of the file is sequential calls to `iq.step_1_*` through `iq.step_5_*`. Each step is a single function call with its own named output dataframe.

See [The Pipeline](/descriptive-systems/the-pipeline) for what each step does.

## iq.py

Statistical logic. The five-step pipeline functions live here:

- `step_1_build_observation_df` — current quarter observations tagged with schema metrics
- `step_2_build_sample_population_dfs` — historic population grouped by context lens, mean/SD and survival distribution
- `step_3_build_observations_with_all_lineage_ids` — observation requisitions expanded across qualifying context lenses
- `step_4_add_z_scores_from_sample_population` — z-scores per observation per context lens
- `step_5_add_survival_from_sample_population` — survival percentiles per observation per context lens

Each step takes the dataframes from previous steps as inputs and produces a named output dataframe. No hidden state. No persistent variables. Every transformation visible.

## etl_functions.py

Reusable ETL primitives. The functions here are domain-agnostic — they operate on dataframes with the right shape regardless of what the data represents.

Core functions:

```python
add_lineage_columns(df, lineage_cluster_name, lineage_cluster_group,
                    lineage_columns, lineage_column_level,
                    sort_column, top_n, lineage_target_column)
```

Adds the `lineage_*` metadata columns to a dataframe. Tracks data provenance through the pipeline.

```python
new_group_by_top_n_columns_df(df, on_columns, sort_column,
                              top_n, ascending_equals, unique_id)
```

Filters a dataframe to groups meeting the `top_n` threshold, then samples `top_n` records per group.

```python
new_mean_sd_df(df, target_column, lineage_unique_id)
```

Adds mean and standard deviation columns to the dataframe based on grouping by `lineage_unique_id`.

```python
add_z_score_to_df(df, target_column, mean_name="mean", sd_name="sd", column_name="z_score")
```

Standardizes the target column relative to group mean/SD. Returns z-scores.

```python
new_lineage_df_group_by_cumulative_count_running_percent(df, target_column, lineage_unique_id)
```

Builds the distribution of `target_column` values per `lineage_unique_id` group, with cumulative percent.

```python
classify_lineage_df_survival_quarter_bins(df, percent_column, lineage_target_column)
```

Bucketizes cumulative percent into 25th/50th/75th/100th survival bins.

Lineage column utilities:

```python
get_lineage_column_names(df)              # returns lineage_* column names
get_columns_starting_with(df, prefix)     # returns columns starting with prefix
drop_lineage_columns_except(df, keep)     # drops all lineage_* except specified
keep_columns_only(df, keep_columns)       # keeps only specified columns
```

These functions handle the bookkeeping of which lineage columns travel through which transformations.

## Configuration Files Cross-Reference

See [Schemas & Configs](/descriptive-systems/schemas-configs) for the full YAML contents.

Each configuration file is itself documentation. Reader can inspect the YAML to understand:

- Which metrics the pipeline evaluates
- Which context lenses qualify
- What signal thresholds apply
- What floor/ceiling boundaries exist
- How survival percentiles are bucketed

The configuration files are the methodology made explicit. Nothing is hardcoded in the pipeline logic that isn't surfaced in the YAML.

## What This Codebase Is

A reference implementation of the Descriptive System. Streamlit-based for prototype operability. Python-based for inspectability and transparency.

Reader can clone the codebase, point it at their own filled requisition data, swap in their own schemas and metric configs, and run the pipeline against their own peer groups.

## What This Codebase Is Not

This codebase is not the production deployment of the Descriptive System. It's the prototype that proved the methodology works.

Production deployment of the Descriptive System happens through the BlueCube framework's broader implementation pattern — the methodology's outputs land as custom fields on requisition records in any ATS, via the same CSV-based composition pattern that drives the Decision Systems.

The reference implementation lives at ReqSquared as part of the body of work that established RequisitionIQ as a methodology. The deployment pattern lives in the BlueCube framework documentation.
