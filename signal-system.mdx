Signal levels are schema-driven. Each metric defines its own z-score thresholds, direction, floor, and ceiling in `schema_filled_reqs.yml`.

## Default Signal Thresholds

| Signal | Threshold | Meaning |
|--------|-----------|---------|
| **Normal** | z < 0.85 | Within expected range |
| **Watch** | z < 1.15 | Early deviation |
| **Monitor** | z < 1.96 | Significant deviation |
| **Inspect** | z > 1.96 | Requires attention |

These are defaults. Each metric in the schema can override them.

## Direction Matters

Lower time-to-fill is good. Higher percent-reviewed is good. The schema specifies the direction per metric.

```yaml
# schema_filled_reqs.yml

time_to_fill:
  dtype: float64
  description: "Days from open to filled"
  direction: -1                    # lower is better
  floor: 15
  ceiling: 90
  z_score_normal: 0.85
  z_score_watch: 1.15
  z_score_monitor: 1.96
  z_score_inspect: 2.5
  survival_25th: 1
  survival_50th: 0.5
  survival_75th: 0
  max_metric_value: 150

percent_reviewed:
  dtype: float32
  description: "Percent of total applicants that have been reviewed"
  direction: 1                     # higher is better
  floor: 0.50
  ceiling: 1
  z_score_normal: 0.85
  z_score_watch: 1.15
  z_score_monitor: 1.96
  z_score_inspect: 2.5
  survival_best: 75th
  survival_25th: 0
  survival_50th: 0.5
  survival_75th: 1
```

Direction is `-1` when lower is better. Direction is `1` when higher is better. The signal interpretation accounts for direction so the *Inspect* level always means *requires attention*, regardless of which direction the metric moved.

## Floor and Ceiling

Each metric has operational boundaries:

- **Floor** — the lower acceptable bound for the metric
- **Ceiling** — the upper acceptable bound for the metric

These are independent of z-score thresholds. A metric value can be within the z-score normal range but still outside the operational floor/ceiling. Both signals matter.

For `time_to_fill`:

- Floor: 15 days (anything below is suspect — may indicate data quality issues)
- Ceiling: 90 days (anything above is operationally concerning regardless of peer group)
- `max_metric_value`: 150 days (hard cap for outlier handling)

For `percent_reviewed`:

- Floor: 0.50 (anything below 50% review rate is operationally concerning)
- Ceiling: 1.00 (the natural upper limit)

## Per-Metric Schema Configuration

Every analytical metric in the engine has its own schema entry. Each entry defines what makes a normal value, a watch value, a monitor value, and an inspect value for that metric specifically.

```yaml
applicants_total:
  dtype: int64
  description: "Total number of applicants"
  direction: 0                     # neither direction is inherently good
  floor: 5
  ceiling: 100
  z_score_normal: 0.85
  z_score_watch: 1.15
  z_score_monitor: 1.96
  z_score_inspect: 2.5
  survival_25th: 0
  survival_50th: 0.5
  survival_75th: 1
```

`applicants_total` uses `direction: 0` because high or low applicant counts both signal different operational realities. Neither is inherently better. The signal is the deviation itself, interpreted by the operator.

```yaml
days_to_first_screen:
  dtype: float64
  description: "Days from open to first screening"
  direction: -1                    # lower is better
  floor: 0
  ceiling: 14
  z_score_normal: 3                # wider tolerance — early process noise
  z_score_watch: 3.5
  z_score_monitor: 4
  z_score_inspect: 4.5
  survival_25th: 1
  survival_50th: 0.5
  survival_75th: 0
```

`days_to_first_screen` uses wider z-score thresholds because early-process variance is higher and the metric distribution is more volatile. The schema accounts for the metric's specific statistical character.

## Survival Percentiles

Beyond z-scores, the engine produces survival percentile bins. Each metric's schema entry includes `survival_*` configuration:

```yaml
time_to_fill:
  survival_25th: 1
  survival_50th: 0.5
  survival_75th: 0
```

For `time_to_fill` (where lower is better), being in the 25th percentile is best (`1`), 75th percentile is worst (`0`). The survival configuration aligns with the direction.

For `percent_reviewed` (where higher is better):

```yaml
percent_reviewed:
  survival_25th: 0
  survival_50th: 0.5
  survival_75th: 1
```

The 25th percentile is worst (`0`), 75th percentile is best (`1`). Same survival mechanic, inverted to match the metric's direction.

## Why Schema-Driven Signals

Hardcoded thresholds break when applied to different organizations, geographies, or job types. The schema-driven approach lets the engine adapt:

- Different organizations have different operational baselines
- Different metrics have different statistical character
- Different periods may require different threshold tightening
- Different audiences may want different signal sensitivity

Change the schema. Change the signals. The engine logic stays the same. The operational configuration travels with the schema YAML.

This is what makes the Descriptive System portable. The pipeline doesn't bake assumptions about what *normal* looks like. The schema does. And the schema is a configuration file that travels with the deployment.

## Signal Composition Across Lenses

A single requisition produces multiple z-scores — one per qualifying context lens. Each context lens produces its own signal level for the same metric.

This is the foundation of the Peer Group Voting System. A requisition might score:

- *Normal* against recruiter context
- *Watch* against country context
- *Monitor* against company context

Three lenses. Three different signal levels for the same metric on the same requisition. The operator sees the composition and decides how to weight the evidence.

The Descriptive System doesn't decide. It surfaces the multi-lens signal stack. The decision lives downstream — in the Decision Systems or in the operator's judgment.
